## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → form → Netlify function | Form data (name, company, project description, timeline) crosses from browser to Netlify serverless function via POST |
| Site → Calendly | Calendly link opens in a new tab; no data is passed in the URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Spoofing | Calendly external link | accept | `rel="noreferrer"` prevents window.opener access. URL is hardcoded (no user-controlled redirect). |
| T-04-03-02 | Tampering | Lead form submission payload | mitigate | Netlify function (existing) receives `name`, `company`, `project`, `timeline`. No server-side auth; function should validate field lengths. Phase 4 does not change the Netlify function — the existing submission path is preserved unchanged. |
| T-04-03-03 | Information Disclosure | form-status error message | mitigate | Error message shows generic text + Gmail fallback. No server error details are surfaced to the user (catch block consumes the error object). |
| T-04-03-04 | Denial of Service | Form submit spam | accept | No rate limiting in scope for Phase 4. Netlify has built-in submission limits. Addressing in Phase 5 or v2 if needed. |
| T-04-03-05 | Information Disclosure | Footer email address | accept | `manikmalhotra6@gmail.com` is intentionally public contact information. Obfuscation adds friction for users, not bots. |
</threat_model>
